# Тестовое задание для позиции Flutter Middle

## Описание проекта

Необходимо реализовать мобильное приложение **Candidate Dashboard** — приложение для HR-специалиста, которое позволяет просматривать список кандидатов, анализировать их соответствие вакансии, управлять статусами обработки резюме и работать в офлайн-режиме.

Задание демонстрирует типичные задачи Flutter-разработчика в экосистеме Түндүк: работа с REST API, управление состоянием, локальное хранилище и офлайн-кэширование, навигация, тестирование.

---

## Технические требования

### Стек
- **Flutter** (актуальная стабильная версия) + **Dart** (null-safety)
- **State management**: BLoC/Cubit **или** Riverpod (на выбор)
- **Навигация**: `go_router`
- **DI**: `get_it` + `injectable` (или аналог)
- **Сеть**: `dio` (+ `retrofit` по желанию) — работа с моком REST
- **Модели**: `freezed` + `json_serializable`
- **Локальное хранилище**: `hive` / `drift` / `shared_preferences` (на выбор)
- **Тесты**: `flutter_test`, `bloc_test`, `mocktail`/`mockito`

### Данные
Реальный backend поднимать не нужно — используйте моки.

- **Основной мок** — `mock/candidates.json` (25 кандидатов, покрывает задания 1–5)
- **Большой мок** — `mock/candidates-large.json` (120 кандидатов, для задания 6 — производительность)

Загрузку данных оформите через `dio` так, чтобы её легко было заменить на реальный HTTP-эндпоинт: чтение мока из assets или локальный mock-interceptor с искусственной задержкой (300–800 мс), имитирующей сеть.

## Задания

### Задание 1. Список кандидатов (главный экран)

**Требования:**
- Отобразить кандидатов в виде списка карточек (`ListView.builder`)
- Для каждого кандидата показать: ФИО, город, общий опыт, вердикт (цветовая индикация), основной стек
- **Фильтрация по вердикту** (Все / ПОДХОДИТ / ЧАСТИЧНО / НЕ ПОДХОДИТ) — chips или сегментированный контрол
- **Поиск по ФИО** с debounce 300 мс
- **Сортировка** по имени, опыту, дате добавления (bottom sheet или меню)
- Подгрузка постранично (pagination / infinite scroll, 10 кандидатов на страницу)
- Состояния: loading (скелетоны/спиннер), пустой результат, ошибка загрузки (с retry)

**Критерии оценки:**
- [ ] Корректная работа фильтрации и поиска
- [ ] Debounce реализован без лишних запросов/перестроений
- [ ] Состояния loading / empty / error обработаны
- [ ] Список плавно прокручивается (без перестроения всего списка)

---

### Задание 2. Карточка кандидата (детальный экран)

**Требования:**
- Переход на `/candidate/:id` через `go_router` (с поддержкой deep link)
- Полная информация: контакты, опыт работы (список), образование, стек (chips)
- Блок **«Критерии оценки»** с цветовой индикацией (ok/warn/no)
- Блок **«Вопросы для собеседования»**
- Блок **«Summary»**
- Контакты кликабельны: телефон (`tel:`), email (`mailto:`), Telegram (`url_launcher`)
- Корректная обработка несуществующего `id` (экран «не найдено»)

**Критерии оценки:**
- [ ] Корректная навигация и передача параметров через go_router
- [ ] Обработка отсутствующего кандидата
- [ ] Разделение на переиспользуемые виджеты (не монолитный build)
- [ ] Корректная цветовая индикация verdict/criteria

---

### Задание 3. Управление статусами

**Требования:**
- В детальной карточке — смена статуса обработки:
  `Новый` → `На рассмотрении` → `Приглашён` → `Отклонён`
- При смене — mock API запрос (имитация `PATCH /candidates/:id/status` с задержкой и случайной ошибкой ~10%)
- **Оптимистичное обновление** UI с откатом при ошибке
- Статус отображается в списке кандидатов (бейдж)
- Loading-индикатор на время запроса, snackbar об успехе/ошибке

**Критерии оценки:**
- [ ] Оптимистичное обновление с корректным откатом при ошибке
- [ ] Состояние статуса консистентно между списком и деталями
- [ ] Обработка ошибок и индикация процесса

---

### Задание 4. Офлайн-режим и кэширование (мобильная специфика)

**Требования:**
- Кэшировать список кандидатов в локальное хранилище (Hive/Drift/...)
- При старте без сети — показывать данные из кэша + индикатор «офлайн»
- Изменённые статусы (Задание 3) сохраняются локально и переживают перезапуск приложения
- Pull-to-refresh для обновления данных при наличии сети
- (Опционально) очередь несинхронизированных изменений статусов с досылкой при появлении сети

**Критерии оценки:**
- [ ] Список доступен в офлайне из кэша
- [ ] Локально сохранённые статусы переживают перезапуск
- [ ] Понятная индикация офлайн-состояния
- [ ] Нет падений при отсутствии сети

---

### Задание 5. Тестирование

**Требования:**
- **Unit-тесты**: логика фильтрации/сортировки; Cubit/BLoC переходы состояний (через `bloc_test`)
- **Тест репозитория**: с замоканным `dio`/datasource (`mocktail`)
- **Widget-тест**: экран списка (отрисовка карточек, фильтр по вердикту, пустое состояние)
- Проверить граничные случаи: пустой список, ошибка загрузки

**Критерии оценки:**
- [ ] Покрыты ключевые состояния BLoC/Cubit
- [ ] Сетевые вызовы замокированы
- [ ] Есть хотя бы один widget-тест
- [ ] Проверены граничные случаи

---

### Задание 6. Производительность и шлифовка (дополнительно)

**Требования:**
- Переключиться на `mock/candidates-large.json` (120 кандидатов)
- Список из 120 элементов должен прокручиваться плавно (`ListView.builder`, `const`-виджеты, без тяжёлых перестроений)
- Минимизировать ненужные перерисовки (`select`/`buildWhen`/`context.select`)
- Поддержка светлой и тёмной темы
- Адаптивная вёрстка (телефон + планшет), корректная работа с `SafeArea`

**Критерии оценки:**
- [ ] Плавная прокрутка большого списка
- [ ] Отсутствие лишних перерисовок (обоснованно)
- [ ] Тёмная тема и адаптивность

## Что оценивается

### Обязательно
- Чистота и читаемость кода, адекватная структура и разделение ответственности
- Корректная типизация и null-safety (минимум `dynamic`, модели через freezed)
- Грамотная работа с состоянием (без «божественных» виджетов и прокидывания состояния руками)
- Обработка ошибок и состояний загрузки
- Работа в офлайне (Задание 4)

### Будет плюсом
- Чистая архитектура и разделение data/domain/presentation
- Покрытие тестами ключевой логики
- Аккуратный UI/UX: тёмная тема, адаптивность, анимации переходов
- Очередь синхронизации статусов в офлайне
- Комментарии к нетривиальным участкам кода

## Уровневая шкала

| Критерий | Junior | Middle | Strong Middle |
|----------|--------|--------|---------------|
| **Код** | Работает, но есть дубли, логика в виджетах | Чистый, переиспользуемые виджеты | Архитектурно грамотный, SOLID |
| **Dart/типизация** | `dynamic`, ручной JSON-парсинг | Полная типизация, freezed/json_serializable | Sealed-классы, обобщения, чистые модели |
| **State management** | `setState` везде | BLoC/Cubit или Riverpod, чистые состояния | Оптимизация перерисовок, селекторы |
| **Сеть/офлайн** | Только загрузка по сети | Кэш + офлайн-режим | Sync-очередь, конфликт-резолюшн |
| **Тесты** | Нет или 1–2 | Покрытие BLoC и репозитория | Widget + граничные случаи, >70% логики |
| **UI/UX** | Базовая вёрстка | Состояния, адаптивность | Тёмная тема, анимации, шлифовка |
